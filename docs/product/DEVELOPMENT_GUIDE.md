# WePicTool 开发指南

**用途：** 开发全程参考，从给 AI 下任务到真机验收
**配套文档：** `WePicTool_PRD.md`、`WePicTool_TECHNICAL_SPEC.md`

---

## 1. 固定任务模板

每次给 AI 编码助手下任务时，使用以下固定结构：

```text
1. 【背景】当前处于哪个阶段，已有能力是什么
2. 【本次任务】只实现本阶段目标
3. 【技术栈】原生微信小程序 + CloudBase + 前端 Canvas
4. 【约束】引用 PRD 开发红线
5. 【验收标准】列出真机要验证的动作
```

### 使用原则

- 一次只做一个阶段。
- 每阶段完成后先真机验收，再进入下一阶段。
- 不要把其他商品、头像、包/配饰、日常合集、表情包、心情状态图提前加进来。
- 不要让 AI 自己决定分类标签和输出比例，按 PRD 来。
- 保存和分享能力必须真机验证。
- 埋点最晚阶段五必须加入，否则无法判断产品是否成立。

## 3. 当前阶段任务口径

```text
实现穿搭白底卡片生成链路。

需求：
1. 前端接收云函数返回的抠图结果（matted/ 目录下的透明主体图）
2. 使用 Canvas 合成白底卡片：
   - 支持 1:1、4:5、3:4 三种比例
   - 背景纯白
   - 主体按组使用视觉锚点：上衣偏中上、下装偏中下、鞋子居中偏下
   - 主体最长边占画布 78%-86%
   - 四周留白统一
   - 白色 / 浅色衣物自动加轻阴影、细描边或切换浅灰底
3. 结果可保存到相册
4. 支持单张保存和按组保存

异常处理：
- 抠图失败的图片保留原图，不影响整批结果
- 失败图显示"重做"按钮

约束：
- 白底卡片合成在前端 Canvas 完成（CloudBase 不支持 sharp 等原生模块）
- MVP 不拍死默认比例，结果页必须允许用户在 1:1 / 4:5 / 3:4 之间切换
- 浅色衣物可见性增强是 P0
- 不加水印
```

## 4. 阶段历史任务口径（归档）

### 4.1 阶段一：跑通最小闭环（不带 AI）

```text
我在做一个微信小程序，当前任务是跑通穿搭参考图的最小技术闭环。

需求：
1. 首页有主按钮"做穿搭参考图"
2. 调用 wx.chooseMedia，支持选 1-9 张图片
3. 上传前本地压缩，最长边不超过 1600px
4. 将图片上传到 CloudBase 云存储
5. 调用 processOutfit 云函数创建处理任务
6. 云函数先不做真实 AI，只把原图按 mock 分类返回
7. 前端展示分组结果页
8. 结果页提供"保存单张"和"保存当前组"

约束：
- 不做登录
- 不做真实抠图
- 不做复杂 UI
- 保存相册必须处理授权被拒
```

### 4.2 阶段二：接入图片部件识别

```text
在已有最小闭环基础上，加入穿搭部件识别能力。

需求：
1. 云函数拿到图片后调用 DashScope qwen-vl-plus 分类
2. 返回 type 和 confidence
3. confidence >= 0.8 直接分组
4. confidence < 0.8 交给用户确认
5. 结果页允许用户修改分类

约束：
- 完整人物试穿图默认 daily
- 头像、包、配饰等非 MVP 素材不进入穿搭主链路
- 分类结果必须可以被用户手动覆盖
```

## 5. 每次开发前检查

1. 当前任务是否只属于一个阶段。
2. 是否引用了 `WePicTool_PRD.md` 的开发红线。
3. 是否有真机验收动作。
4. 是否避免把日常合集、表情包、心情状态图提前放入主入口。
5. 是否保留每组不足 3 张时的降级提醒。

## 7. 验收标准模板

### 阶段三验收标准

- [ ] 用 20 张穿搭素材测试，白底结果可接受率 >= 70%。
- [ ] 白色 / 浅色衣物在白底上仍然可辨认。
- [ ] 单张处理端到端 <= 10 秒。
- [ ] 3 张处理端到端 <= 20 秒。
- [ ] 失败图不影响其他结果展示。
- [ ] 按组保存真机通过。
- [ ] 1:1 / 4:5 / 3:4 三种比例切换真机可用。
- [ ] iOS + Android 真机各跑一遍。

---

## 8. 开发时最应该先验证的技术点

1. 小程序 1-9 张图片上传是否稳定。
2. 批量保存图片是否打扰用户，失败是否可重试。
3. 抠图 API 对衣服边缘、蕾丝、浅色衣服、鞋带的效果。
4. 多模态分类能否稳定识别上衣、下装、鞋子，并把头像/包/配饰识别为非 MVP 类别。
5. 1:1 / 4:5 / 3:4 哪个比例在微信聊天流里观感舒服。
6. 微信分享图片菜单和保存相册在真机是否稳定。
7. 用户是否能理解"按组保存后回微信发送"的路径。
8. 分类后每组不足 3 张时，用户是否理解普通发送/补图提醒。
9. 低置信度分类弹窗是否增加负担。
10. 前端 Canvas 合成大图时 iOS/Android 内存是否稳定。

## 10. 已知限制

| 限制 | 说明 | 应对 |
|------|------|------|
| CloudBase 不支持 sharp | 原生 C++ 模块错误码 145 | 改用前端 Canvas 合成 |
| 抠图使用生成模型 | qwen-image-2.0 非专业分割 API | 边缘质量可能不稳定，保留原图兜底 |
| 小程序无法直发微信聊天 | 微信能力限制 | 保存 + 发送引导 |
| `private_getBackgroundFetchData:fail` | 微信开发者工具/后台拉取相关提示 | 不是当前业务代码主动调用造成，可忽略 |
