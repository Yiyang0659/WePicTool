## 1. 平台事实（复刻效果的物理基础）

以下每条都是**已核实的微信平台行为**（核实日期 2026-07-18），是所有玩法设计的前提，不随我们的实现改变：

| # | 平台事实 |
| --- | --- |
| 1 | 在微信聊天中发送 **3 张及以上** 图片/视频时，发送面板出现「发送后合并展示」选项，**需用户手动勾选**，默认不勾选。 |
| 2 | 合并发送后以**折叠卡片**展示：有封面、可左右滑动预览、可点击展开/收起。 |
| 3 | 支持**一键保存全部**、**整体转发**。 |
| 4 | 单次最多发 **99 张**。 |
| 5 | **视频可混入叠图**（15 秒内小视频，或相册 5 分钟内视频）。 |
| 6 | 合并展示仅 **iOS / Android 生效**，电脑端仍逐张显示；无漏看提醒。 |

### 1.1 关键推论：封面和顺序是工具唯一能设计的东西

- 叠图的**封面**和**滑动顺序**由用户在微信里的**勾选顺序**决定，小程序无法控制微信侧行为。
- 因此工具能设计的只有两件事：**每张卡的内容**，和**保存到相册的文件顺序**。
- 推论：**保存文件名必须带编号（01、02……）**，且**发送引导必须教用户按编号顺序勾选**。顺序控制是命根子。

### 1.2 ≥3 张硬约束

- 不足 3 张不触发「发送后合并展示」选项，玩法整体不成立。
- **所有玩法必须保证最少产出 3 张卡**；用户输入天然不足 3 张时，由管线自动补封面卡/引导卡凑满（见第 3 章兜底策略与各模块实现卡）。

## 2. 产品定位与红线

### 2.1 定位

微信「合并发送 / 叠图」玩法生成器——**穿搭白底卡是旗舰功能，玩法模板库是长期资产**。

> 一叠图 = 有封面、有顺序、有节奏、可整体流通的图组。

- 不做衣橱管理，不做通用表情包工具。
- 核心资产 = **玩法模板库** + **对叠图交互细节的理解**（封面、顺序、节奏、展开双形态）。

### 2.2 红线

1. 所有新玩法必须走**统一叠图管线**（第 3 章），不做独立形态的功能。
2. 整蛊类玩法只做**可爱反转**，不做恐怖惊吓内容，避免投诉与审核风险。
3. 用户输入文字必须过 `msgSecCheck` **内容安全检测**。
4. 结果图**不加水印**。
5. **每叠最少 3 张卡**（不足时自动补卡，不得产出可发送但无法合并展示的图组而不提示）。

## 3. 统一叠图管线

所有模块共享同一条 6 段管线。新玩法 = 新增输入器 + 卡片生成器（注册一个新玩法模板），后半段全部复用：

```text
输入器（选图 / 文字 / 模板参数）
  → 卡片生成器（前端 Canvas 为主）
  → 叠图预览（复用 pages/preview 微信聊天预览页）
  → 编号保存（文件名 01、02……控制发送顺序，复用现有 saveImagesSequentially）
  → 发送引导（教用户按编号勾选 + 勾选「发送后合并展示」）
  → 回流引导卡（末卡"用 WePicTool 做同款"，可开关，埋点单独统计）
```

### 3.0 用户交互主流程图

```mermaid

    B -->|工具区| E[资料打包<br/>多张照片分组]
```

读图要点：
- F→G 之后是发送方能控制的全部（每张卡内容 + 文件顺序）；M 之后是微信侧行为，工具不可控，只能引导。
- O→A 是裂变回流环：收图人转化为新用户的唯一设计抓手。

### 3.1 玩法模板注册表

每个玩法是一个注册项：

```js
{
  id: 'big-text',              // 玩法唯一 ID（埋点维度）
  name: '大字滑卡',
  inputType: 'text',           // text | images | template-params | mixed
  composeFn: 'composeBigText', // 卡片生成函数（前端 Canvas）
  cardCountRule: '每张 1 个字，不足 3 张自动补封面/引导卡',
  minCards: 3,                 // ≥3 张硬约束
  fallbackStrategy: '补封面卡 + 引导卡凑满 3 张',
  trackDimension: { moduleId: 'big-text', templateId: null }
}
```

新玩法 = 在注册表新增一个注册项，接入管线既有组件。

### 3.2 通用组件清单（底座能力）

| 组件 | 说明 | 现状 |
| --- | --- | --- |
| 编号保存 | 按 01、02…… 文件名顺序保存到相册 | 已有：`pages/result` 的 `saveImagesSequentially`，需抽为通用组件并加文件名编号 |
| 发送引导浮层 | 教用户"按编号勾选 + 勾选「发送后合并展示」" | 已有发送引导，需按叠图口径改版为通用浮层 |
| 回流卡生成器 | 生成末卡"用 WePicTool 做同款"，可开关 | 新增（管线第 6 段） |
| msgSecCheck 接入点 | 用户输入文字的内容安全检测 | 新增，放云函数侧统一封装 |

### 3.3 埋点体系

在原 8 个事件（`task_created`、`classification_completed`、`category_changed`、`task_completed`、`group_saved`、`result_saved`、`share_guide_clicked`、`retry_triggered`）基础上**新增 6 个**：

| 事件 | 说明 |
| --- | --- |
| `module_entered` | 进入某功能模块（带模块 ID） |
| `template_used` | 使用某玩法模板（带模板 ID） |
| `stack_saved` | 一叠图保存完成（带张数、玩法类型） |
| `send_guide_completed` | 看完发送引导并去发送 |
| `reflow_card_impression` | 回流引导卡随叠图生成/预览曝光 |
| `reflow_card_toggled` | 用户打开/关闭回流引导卡开关 |

**按玩法维度看漏斗：** `module_entered` → `template_used` → `stack_saved` → `send_guide_completed` → 分享转化，决定模板库加码方向。回流卡单独统计曝光与开关，验证"收到图的人转化为新用户"的裂变链路。

## 4. 模块实现卡

首页分三层：**主入口（旗舰功能）→ 玩法区 → 工具区**。

| 模块 | 一句话说明 | 是否依赖 AI | 建议阶段 |
| --- | --- | --- | --- |
| 穿搭叠图（已有） | 拍衣物 → 白底卡 → 分组叠图发送 | 是（分类 + 抠图） | 已实现待真机验收 |
| 大字滑卡 | 输入一句话，滑一下出一个字 | 否 | 阶段六 |
| 剧情滑卡 | 整蛊/表白/生日模板，滑到最后有反转 | 否 | 阶段六 |
| 盲盒抽卡 | 聚会抽签、答案之书，滑停在哪张算哪张 | 否 | 阶段七 |
| 拼图揭秘 | 大图切块，封面悬念，展开拼全图 | 否 | 阶段七 |
| 资料打包 | 多张照片分组多叠 + 封面目录卡，防刷屏 | 否 | 阶段七 |
| 翻页动画 | GIF/视频抽帧，滑一滑就"播放" | 弱（抽帧为主） | 阶段八 |
| 成套搭配 | 一套穿搭一叠，朋友回复"第 2 套" | 复用现有 | 阶段八 |
| 滑滑换装 | 封面人台固定，每滑一张换一套衣服 | 是（试穿/叠加） | 阶段九 |

每张卡按固定 6 段结构编写：① 对标效果 ② 输入 → 输出 ③ 收图人体验 ④ 技术实现路径 ⑤ 依赖与风险 ⑥ 验收标准。

### 4.1 穿搭叠图（旗舰）

**① 对标效果：** 微信官方文章"研究穿搭"演示——白底分组卡片叠图，朋友左右滑动对比上衣/下装/鞋。官方示例已验证白底分组卡片是叠图最有质感的用法，但用户自己拍的照片背景杂乱，直接发不出手。

**② 输入 → 输出：** 用户选 1–9 张衣物照 → AI 自动分类上衣/下装/鞋 → 抠图换白底 → 按组生成白底卡（支持 1:1 / 4:5 / 3:4，主体锚点分组差异化）→ 每组为一叠，编号保存。封面 = 每组第 1 张（编号 01），顺序 = 组内序号。

**③ 收图人体验：** 先收到上衣组叠图（白底卡逐张滑），再下装组、鞋子组；每组封面即第一件单品，滑动节奏 = 逐件对比，可回复"上衣 1 + 下装 2"。

**④ 技术实现路径：** 已实现。`wx.chooseMedia` → 前端压缩 → 云存储 → 云函数 `processOutfit`（qwen-vl-plus 分类 + qwen-image-2.0 抠图）→ 前端 Canvas 白底合成（`utils/cardComposer.js`）→ 保存相册（`saveImagesSequentially`）→ 发送引导。

**⑤ 依赖与风险：** 抠图边缘质量（生成模型非专业分割，保留原图兜底）；浅色衣物白底可见性（轻阴影 + 细描边已实现，真机验证）；前端 Canvas 大图内存（iOS/Android 待验证）。

**⑥ 验收标准：** 状态为**已实现待真机验收**，引用 `DEVELOPMENT_GUIDE.md` 阶段三验收标准逐项过：

- 用 20 张穿搭素材测试，白底结果可接受率 >= 70%。
- 白色 / 浅色衣物在白底上仍然可辨认。
- 单张处理端到端 <= 10 秒；3 张 <= 20 秒。
- 失败图不影响其他结果展示。
- 按组保存真机通过；1:1 / 4:5 / 3:4 三种比例切换真机可用。
- iOS + Android 真机各跑一遍。
- 叠加定位口径：真机勾选「发送后合并展示」后与官方"研究穿搭"演示逐项对比（封面、滑动、展开/收起、一键保存全部）。

### 4.2 大字滑卡

**① 对标效果：** 微信官方宣传片同款逐字卡（"试试看这个功能人人都能用了"式）——一叠白底/纯色卡，每张一个大字，滑一下出一个字。

**② 输入 → 输出：** 用户输入一句话（建议 ≤ 20 字）→ 选配色模板（白底黑字 / 黑底白字 / 应援色等 3–5 套）→ 每个字生成一张卡。**输入少于 3 个字时，自动补封面卡 + 引导卡凑满 3 张**（兜底策略，见红线 5）。封面 = 第 1 个字（或封面卡），顺序 = 语句顺序。

**③ 收图人体验：** 滑一下出一个字，节奏逐字推进，最后一张停在最想强调的字上；纯文字即有传播力。

**④ 技术实现路径：** 纯前端 Canvas 文字渲染（单字占卡面约 80%、居中），3–5 套配色模板；无需云函数、无 AI 成本；文字接 `msgSecCheck`（云函数侧统一封装）；编号保存与发送引导复用管线。约 1–2 天工作量。

**⑤ 依赖与风险：** 文字内容必须过 `msgSecCheck`；超 20 字截断或提示（叠图节奏衰减）；无 spike 项。

**⑥ 验收标准：** 真机输入 3–20 字 → 保存 → 回微信按编号勾选 + 勾选「发送后合并展示」→ 与官方宣传片逐字卡对比滑动节奏一致；输入 1–2 字时自动补满 3 张且发送引导提示；不当内容被 `msgSecCheck` 拦截并提示。

### 4.3 剧情滑卡

**① 对标效果：** 微信官方"整蛊朋友"演示（13 张"右滑 → 再滑 → 出去。"）——逐张节奏引导 + 末尾反转。

**② 输入 → 输出：** 用户选模板（整蛊 / 表白 / 生日倒数 / 道歉反转）→ 填入称呼或名字、可上传最后一张图 → 生成 8–13 张节奏卡 → 编号保存。封面 = 模板第 1 张（悬念句），顺序 = 模板定义的节奏序列。

**③ 收图人体验：** 封面勾起好奇 → 每张一句短句，停顿节奏由模板控制 → 末张反转。反转结构自带转发——收到的人一定会问"这怎么做的"，是最强的裂变引擎。

**④ 技术实现路径：** 做模板 JSON 引擎：每个模板是一个 JSON（每张卡的文案、停顿节奏、末图位置），前端 Canvas 批量渲染。首批 5–8 个模板；模板引擎 2–3 天，之后每个新模板约半天；用户填写的文字过 `msgSecCheck`；埋点按模板 ID 统计，数据驱动模板迭代。

**⑤ 依赖与风险：** 整蛊只做可爱反转，不做恐怖惊吓（红线 2）；用户自定义称呼过安全检测；模板文案需审核把关，避免擦边内容。

**⑥ 验收标准：** 真机用每个模板生成 → 发送 → 与官方"整蛊朋友"13 张演示对比节奏结构（封面悬念 → 逐张引导 → 末张反转）一致；自定义称呼出现在正确卡位；末图（如上传）位置正确；埋点 `template_used` 带模板 ID 上报。

### 4.4 盲盒抽卡

**① 对标效果：** 叠图"滑停"动作即抽签——一叠卡背统一的卡，每个人滑停在哪张算哪张（官方"合并发图"玩法的互动变体）。

**② 输入 → 输出：** 用户选玩法（聚会抽签 / 答案之书 / 今日运势）→ 设置选项数量和内容（或用预设）→ 生成卡背统一的一叠 → 发到群里。**选项不足 3 个时自动补空卡/引导卡凑满 3 张**（兜底策略）。封面 = 统一卡背，顺序 = 前端打乱后的顺序。

**③ 收图人体验：** 看到一叠卡背，凭感觉滑停在某张，翻开即是自己的"签"；互动即拉新。

**④ 技术实现路径：** 纯前端。统一卡背模板 + 选项卡渲染，顺序在前端打乱；不需要 AI 和云函数；自定义选项文字过 `msgSecCheck`。

**⑤ 依赖与风险：** 自定义选项必须过安全检测；"答案之书"类文案避免迷信敏感表述，走轻松娱乐口径。

**⑥ 验收标准：** 真机生成一叠 → 卡背全部一致 → 滑停在任意张可查看对应选项；选项不足 3 个时自动补满 3 张；打乱顺序后保存编号与滑停结果一致；不当选项文字被拦截。

### 4.5 拼图揭秘

**① 对标效果：** 利用叠图"双形态"——封面只露一块碎片制造悬念，展开后拼出完整大图（对官方折叠卡片展开/收起形态的玩法化使用）。

**② 输入 → 输出：** 用户上传一张大图 → 选切分数量（3 / 6 / 9 块）→ 生成切块卡序列，编号保存。封面 = 第 1 块（一角），顺序 = 展开后的阅读顺序。

**③ 收图人体验：** 封面只看到一角（悬念）→ 滑动逐块浮现 → 展开态拼出全图（惊喜落点）。适合海报预告、宝宝照片、惊喜礼物揭秘。

**④ 技术实现路径：** 纯前端 Canvas 裁切；**关键是切块顺序与编号映射要对应展开后的阅读顺序**（先行后列）；发送引导复用管线。

**⑤ 依赖与风险：** 微信折叠卡片展开态的排版间隙会断开拼图，需要在验收时确认真机展开观感；大图裁切后的单块尺寸不能小到看不清。

**⑥ 验收标准：** 真机 3/6/9 块各生成一叠 → 封面只露一角 → 展开态按阅读顺序能拼回完整大图；编号顺序与展开顺序映射正确；与官方折叠卡片双形态逐项对比。

### 4.6 资料打包

**① 对标效果：** 微信官方自己的卖点"避免照片轰炸"——多叠合并发送 + 每叠封面目录卡，对方一键保存全部。

**② 输入 → 输出：** 用户选 9–50 张照片 → 分组（v1 只做**手动分组**）→ 每组自动生成封面目录卡（标题 + 日期 + 张数）→ 多叠编号保存 → 依次合并发送。

**③ 收图人体验：** 不再被几十张图刷屏；每叠封面目录卡一眼看清"这组是什么、几张"；可一键保存全部。

**④ 技术实现路径：** 分组逻辑 + Canvas 封面目录卡生成；`wx.chooseMedia` 单次上限 9 张，需分批选择；无 AI；可与「记录」页打通复用历史分组。

**⑤ 依赖与风险：**

- **"按时间分组"降级：** `wx.chooseMedia` 拿不到可靠拍摄时间（无 EXIF 回传），**v1 只做手动分组，时间分组标记"待验证"**，不进入首版承诺。
- **分批选择：** 超过 9 张需分多次调用 `wx.chooseMedia`，需做已选列表管理。
- 张数上限受相册保存与发送操作负担约束，v1 上限 50 张。

**⑥ 验收标准：** 真机分批选 20+ 张 → 手动分 3 组 → 每组封面目录卡信息正确（标题 + 张数）→ 多叠保存编号清晰 → 依次合并发送后对方可一键保存全部；不出现"按时间分组"入口（待验证标记仅在文档中）。

### 4.7 翻页动画

**① 对标效果：** 微信官方"隔空一拳"演示（3 帧出拳）——快速滑动时帧序列形成"播放"效果，是官方玩法里视觉冲击力最强、但普通人完全做不了的（抽帧、排序、节奏都太麻烦）。门槛越高，工具价值越大。

**② 输入 → 输出：** 用户上传 GIF 或 3 秒内短视频（或直接选动作素材库：出拳 / 比心 / 变装）→ 抽帧成 8–24 张 → 预览滑动播放效果 → 编号保存。封面 = 第 1 帧，顺序 = 时间正序。

**③ 收图人体验：** 快速滑动时画面"动起来"——一拳挥出、一颗心比出来。**帧数甜点区间 8–12 帧**；上限 24 帧，手动滑 24 帧体验差（手酸、节奏拖沓），不得超出。

**④ 技术实现路径：** 两条路线——

1. **素材库先行（v1）：** 预置若干动作帧序列（出拳 / 比心 / 变装），用户零等待直接用，纯前端。
2. **自定义抽帧（v2）：** 小程序端视频抽帧兼容性差，走**云托管跑 ffmpeg** 抽帧后回传帧序列。云托管是容器制，可跑 ffmpeg，绕开 CloudBase 云函数不支持原生模块的限制（sharp 错误码 145 同因）。

先上①验证玩法，再加②；**路线②上线前需做一次云托管抽帧验证（耗时 / 回传体积 / 费用）**。

**⑤ 依赖与风险：** 路线②是 spike 项：云托管冷启动耗时、帧序列回传体积（8–12 张图的流量成本）、云托管计费模式均需实测；用户上传视频内容需注意审核口径。

**⑥ 验收标准：** 素材库路线：真机选素材 → 生成 8–12 帧 → 快速滑动有明确"播放"感 → 与官方"隔空一拳"对比动效成立。自定义抽帧路线：云托管抽帧验证报告（耗时 / 体积 / 费用）产出后，真机上传 3 秒视频 → 抽帧张数在甜点区间 → 滑动播放流畅。

### 4.8 成套搭配

**① 对标效果：** 穿搭主线自然深化——从"分组发单品"升级为"发成套搭配"，朋友直接回复"第 2 套"（官方"研究穿搭"演示的进阶用法）。

**② 输入 → 输出：** 基于本次上传或历史衣物 → 自动/手动组合 N 套 → 一套一叠（上衣 1 + 下装 1 + 鞋 1）→ 多叠编号保存发送。每叠封面 = 该套第 1 件，顺序 = 上衣 → 下装 → 鞋。

**③ 收图人体验：** 一叠 = 一套完整搭配，滑 3 张看完一套；多叠对比后直接回复"第 2 套"，与阶段四编号角标、投票场景打通。

**④ 技术实现路径：** 复用穿搭管线全部能力（分类 + 抠图 + 白底卡 + 编号保存），增量只有**组合逻辑**（N 件上衣 × M 件下装排列 + 手动调整）和**多叠保存**。**建议等穿搭主链路留存数据稳定后再做。**

**⑤ 依赖与风险：** 单套只有 3 张（上衣 + 下装 + 鞋），恰好满足 ≥3 张硬约束，但某组缺图时（如无鞋）需自动补封面卡凑满；组合数爆炸需限制 N 套上限。

**⑥ 验收标准：** 真机上传 3 上衣 × 2 下装 → 生成多套 → 每叠 ≥3 张（缺组自动补卡）→ 多叠发送后朋友可按叠回复编号；与官方"研究穿搭"演示对比单叠观感一致。

### 4.9 滑滑换装

**① 对标效果：** 滑动交互和换装是天生一对——封面人台固定，每滑一张换一套，像芭比换装（叠图帧序列玩法与穿搭主线汇合的"大杀器"）。

**② 输入 → 输出：** 用户选一个虚拟人台（或上传自己的照片）→ 从衣柜挑几套 → 生成"人台不变、衣服在变"的帧序列 → 发送后朋友滑一滑帮你选。封面 = 人台 + 第 1 套，顺序 = 套数顺序，套数不足 3 时补封面/引导卡凑满 3 张。

**③ 收图人体验：** 人台位置固定不动，滑一下换一套衣服，对比极其直观；最后可以停在最喜欢的一套截图回复。

**④ 技术实现路径：** 依赖抠图质量——衣物白底图叠加到人台对应位置：**上衣锚点偏上、下装锚点偏下，复用现有分组锚点体系**（`cardComposer.js` 的 `GROUP_ANCHOR`）；进阶可接 AI 试穿 API 生成真实上身效果。**放最后做，等阶段三抠图质量验证通过后启动。**

**⑤ 依赖与风险：** 抠图边缘质量直接决定换装观感（spike：阶段三真机验收的抠图质量结论）；上传真人照片有隐私与审核风险，v1 建议只做虚拟人台；AI 试穿 API 的成本与合规待评估。

**⑥ 验收标准：** 真机选人台 + 3 套衣服 → 帧序列人台位置不变、衣服逐套切换 → 滑动换装节奏成立；套数不足 3 时自动补卡；与官方叠图滑动体验对比无明显违和。
